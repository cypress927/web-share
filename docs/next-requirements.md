# 下一阶段需求文档

## 说明

本文档整理下一阶段要实现的体验与功能需求。

当前仅用于需求收敛，不作为本次 Git 提交内容。

## 1. 托盘菜单行为调整

状态：`已完成`

### 目标

将当前托盘菜单中的 `退出托盘` 改为 `退出程序`。

### 预期行为

- 点击 `退出程序` 后：
  - 关闭托盘图标
  - 停止后台管理器
  - 停止当前所有分享任务
  - 释放占用端口

### 补充说明

- 当前仅退出托盘，不退出后台管理器，用户容易误以为程序已经完全关闭
- 修改后应保证行为符合普通用户对“退出程序”的直觉

### 验收标准

- 点击 `退出程序` 后，托盘消失
- 管理页无法继续访问
- 当前分享链接失效
- 再次右键分享时，程序可重新按需启动

## 2. 前端页面增加子文件夹进入功能

状态：`已完成`

### 目标

在文件夹分享页面中，支持进入子文件夹继续浏览，而不仅仅显示根目录内容。

### 预期行为

- 访问文件夹分享页时：
  - 可以点击子文件夹名称进入子目录
  - 可以继续浏览更深层级目录
  - 页面中应有返回上一级入口

### 路径建议

- 分享入口仍然是 `/s/<share-code>`
- 子目录可考虑使用：
  - `/s/<share-code>/browse?path=sub1/sub2`
  - 或 `/s/<share-code>/d/<encoded-path>`

### 安全要求

- 必须防止路径穿越
- 不允许跳出共享根目录
- 所有子路径都必须被约束在当前共享目录内

### 验收标准

- 可连续进入多级子目录
- 可返回上级目录
- 子目录中文件仍可下载
- 无法通过构造路径访问共享根目录外文件

## 3. 下载增强：断点续传

状态：`已完成`

### 目标

支持文件下载断点续传，便于大文件在移动端或不稳定网络环境下下载。

### 预期行为

- 文件下载接口支持 HTTP Range 请求
- 客户端中断后再次下载时，可以继续下载剩余部分

### 实现要求

- 正确返回：
  - `Accept-Ranges: bytes`
  - `206 Partial Content`
  - `Content-Range`
- 对单文件下载和目录中文件下载都生效

### 验收标准

- 浏览器或下载工具可触发分段下载
- 大文件中断后可以继续下载
- 不影响原有普通下载

## 4. 下载增强：整个文件夹打包下载

状态：`已完成`

### 目标

支持将整个共享文件夹打包后下载。

### 预期行为

- 文件夹分享页面中提供：
  - `下载整个文件夹`
- 点击后：
  - 服务端动态打包该文件夹
  - 返回压缩包下载

### 页面入口要求

- 在分享页面中提供一个明确按钮：
  - `下载整个分享内容`
- 该按钮语义表示：
  - 无论当前正在浏览哪个子目录
  - 都可以一键下载当前分享根目录的完整打包内容
- 对访问者来说，这是“下载整个分享”的固定入口，不依赖当前目录位置

### 格式建议

- 优先使用 `.zip`

### 实现要求

- 支持包含子目录的完整打包
- 压缩包名称默认使用分享名称
- 对大目录应考虑流式输出，避免全部先落盘

### 验收标准

- 能下载包含完整层级的 zip 包
- 中文文件名在压缩包内保持正常
- 打包下载不影响普通单文件下载
- 分享页面中始终存在“下载整个分享内容”按钮

## 5. 上传增强：分片上传

状态：`已完成`

### 目标

为文件夹分享页面增加更简单的分片上传能力，提升大文件上传稳定性，同时避免复杂的分片落盘与合并流程。

### 预期行为

- 上传文件时，前端将文件切成多个分片，并按顺序逐个上传
- 服务端收到分片后，按顺序直接追加写入目标文件
- 某个分片上传失败时，只需要重试当前失败分片
- 全部分片成功后，目标文件即上传完成，不再单独执行“合并文件”

### 场景要求

- 适合大文件上传
- 网络波动时不需要整文件重传
- 实现尽量简单，避免额外的上传会话管理复杂度

### 建议实现方案

- 前端在上传开始前先声明：
  - 文件名
  - 文件大小
  - 分片大小
  - 当前分片索引
  - 总分片数
- 服务端校验：
  - 上传密码
  - 文件名合法性
  - 分片索引是否连续
  - 当前已写入偏移是否符合预期
- 服务端写入策略：
  - 首个分片创建目标文件
  - 后续分片按顺序追加写入同一个目标文件
  - 最后一个分片完成后，将该文件视为上传完成
- 失败恢复策略：
  - 允许前端查询当前已接收的字节数或分片序号
  - 前端从最后一个成功分片之后继续上传

### 不采用的方案

- 不做“分片全部落盘后再合并”
- 不做复杂的上传会话目录管理
- 不做服务端并行乱序接收分片

### 安全要求

- 仍受上传密码控制
- 只能写入当前共享目录内的目标路径
- 必须防止利用文件名或路径参数进行路径穿越
- 若上传未完成，服务端需要能识别未完成状态，避免把残缺文件误标记为成功

### 验收标准

- 大文件可以分片上传成功
- 某个分片失败后可以重新上传该分片
- 断点继续上传后文件内容正确
- 服务端无需额外“合并文件”步骤

## 6. 上传增强：上传进度条

状态：`已完成`

### 目标

在前端页面中增加与“顺序分片上传”一致的清晰进度反馈。

### 预期行为

- 选择文件后开始上传时：
  - 显示总体进度条
  - 显示百分比
  - 显示当前状态，例如：
    - 正在上传分片
    - 正在等待服务端写入
    - 上传完成
    - 上传失败
- 进度条应基于“已成功上传的分片 / 已成功写入的字节数”实时推进
- 当前分片失败时，页面应停留在失败状态，并允许继续重试该分片

### 交互建议

- 上传按钮在上传过程中禁用
- 页面中显示：
  - 当前文件名
  - 文件总大小
  - 已上传大小
  - 当前分片序号 / 总分片数
- 上传完成后给出成功提示
- 上传失败后给出失败原因和重试入口

### 验收标准

- 用户能看到实时上传进度
- 顺序分片上传与进度条联动
- 上传完成或失败状态明确

## 7. 前端体验补充建议

### 文件夹浏览

- 当前路径面包屑
- 返回上一级按钮
- 子文件夹与文件在视觉上区分更明显

### 下载入口

- 文件下载与文件夹打包下载分开展示
- 对大文件可显示文件大小

### 上传入口

- 上传中显示文件名、大小、进度
- 上传完成后刷新当前目录列表

## 8. 右键分享时的启动与完成反馈

状态：`已完成`

### 目标

当用户通过右键发起分享时，无论程序当前是否已经启动，都要给用户明确、连续的状态反馈，避免“点了以后没反应”。

### 场景 1：程序尚未启动

#### 预期行为

- 用户右键选择分享后，应立即看到程序正在启动的反馈
- 管理器和托盘启动完成后，应明确提示“程序已就绪”
- 新分享添加完成后，应明确提示“分享已添加”

#### 建议反馈形式

- Windows 通知气泡 / 托盘气泡
- 或轻量级桌面提示

#### 推荐反馈顺序

1. `正在启动 Web Share...`
2. `Web Share 已启动`
3. `分享已添加：<分享名称>`

### 场景 2：程序已经启动

#### 预期行为

- 用户右键选择分享后，不需要再次打开主界面
- 但仍应立即收到“正在添加分享”的反馈
- 添加完成后应明确提示“分享已添加”

#### 推荐反馈顺序

1. `正在添加分享...`
2. `分享已添加：<分享名称>`

### 用户感知目标

- 用户在任何一次右键分享操作后，都能确认：
  - 点击已经生效
  - 程序是否正在启动
  - 分享是否已经添加成功

### 验收标准

- 程序未启动时，右键分享不会表现为静默等待
- 程序已启动时，右键分享也不会表现为无反馈
- 添加成功后，用户能收到包含分享名称的明确提示
- 如果添加失败，也应收到失败提示

## 9. 推荐实现顺序

建议按以下顺序推进：

1. 托盘 `退出程序`
2. 子文件夹进入功能
3. 断点续传下载
4. 整个文件夹打包下载
5. 上传进度条
6. 分片上传

## 10. 风险与注意点

### 路径安全

- 子目录浏览与打包下载都必须严格防止路径穿越

### 性能

- 打包整个文件夹和大文件分片上传都可能增加 CPU、内存、磁盘占用

### 并发

- 分片上传会引入上传会话管理、并发写入和临时文件清理问题

### 兼容性

- 移动端浏览器对大文件下载、上传、进度事件的支持要做实际验证
