# 下一阶段需求文档

## 说明

本文档只保留当前尚未完成、准备下一阶段继续推进的需求。

当前仅用于需求收敛，不作为本次 Git 提交内容。

## 1. 剪贴板分享

### 目标

除文件和文件夹外，支持把当前剪贴板内容直接创建为一个新的分享。

### 范围建议

第一阶段优先支持：

- 剪贴板文本
- 剪贴板图片

后续可扩展：

- 剪贴板文件列表
- HTML / 富文本

### 推荐交互

剪贴板分享不依赖文件系统对象，因此不建议放在文件/文件夹右键菜单中。推荐入口：

- 托盘菜单增加 `分享剪贴板`
- 管理页增加 `新建剪贴板分享`
- 后续可选增加全局快捷键

### 预期行为

#### 文本

- 用户复制一段文本后，点击 `分享剪贴板`
- 程序读取当前剪贴板文本并创建一个只读分享
- 访问页展示格式化文本内容
- 页面提供：
  - 复制文本
  - 下载 txt

#### 图片

- 用户复制一张图片后，点击 `分享剪贴板`
- 程序读取当前剪贴板图片并创建一个只读分享
- 访问页直接预览图片
- 页面提供原图下载

### 数据模型建议

现有分享模型建议扩展 `Kind`，至少支持：

- `file`
- `dir`
- `clipboard_text`
- `clipboard_image`

同时：

- 文件/文件夹继续使用 `Path`
- 剪贴板文本使用 `TextContent`
- 剪贴板图片使用临时文件或二进制对象，并带上 `MimeType`

### 生命周期建议

剪贴板分享采用“快照”模式，而不是实时绑定。

也就是：

- 用户点击分享时读取一次当前剪贴板
- 创建出的分享内容固定不变
- 后续剪贴板变化不影响已创建的分享

### 安全要求

- 剪贴板分享默认只读
- 创建前要有明确反馈，避免误分享敏感内容
- 支持一键停止分享
- 建议后续支持有效期

### 验收标准

- 能从托盘或管理页创建剪贴板分享
- 剪贴板文本可正常查看、复制、下载
- 剪贴板图片可正常预览和下载
- 已创建分享不会跟随后续剪贴板变化而改变

## 2. 前端体验补充优化

### 目标

在现有文件/文件夹分享功能基本可用的前提下，继续优化前端体验细节。

### 建议项

#### 文件夹浏览

- 当前路径面包屑可进一步增强可读性
- 子文件夹与文件的视觉差异可再拉开
- 当前目录操作区域可更聚焦

#### 下载入口

- 文件下载与文件夹打包下载进一步区分视觉层级
- 对大文件展示更明确的文件大小
- 对子文件夹的打包下载入口可继续优化成更直观按钮样式

#### 上传入口

- 上传中展示当前文件名、大小、进度
- 多文件/文件夹上传时显示当前正在处理的文件
- 上传完成后当前目录列表刷新体验可更顺滑

### 验收标准

- 手机和桌面端页面都更容易理解当前可执行操作
- 访问者能快速区分“进入目录”“下载文件”“打包下载目录”
- 上传过程状态比当前版本更清晰

## 3. 推荐实现顺序

建议按以下顺序推进：

1. 剪贴板文本分享
2. 剪贴板图片分享
3. 托盘入口与管理页入口
4. 分享页和管理页的前端体验细化

## 4. 风险与注意点

### 剪贴板兼容性

- Windows 剪贴板格式较多，第一阶段应避免一次支持过多类型
- 图片和文本优先，文件列表可后置

### 安全

- 剪贴板内容通常比普通文件更敏感
- 必须避免“用户误操作后静默公开”

### 存储

- 文本适合直接存内存
- 图片更适合落临时文件，避免占用过多内存

### 前端兼容性

- 移动端对图片预览、文本复制按钮、下载行为都需要实际验证
